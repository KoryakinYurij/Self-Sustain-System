# SOTA Agent Skills & MCP Research Report (March 2026)

Этот отчет содержит рекомендации по развитию проекта `Skill-Factory` на основе актуальных State-of-the-Art (SOTA) технологий в области Agentic Frameworks и Model Context Protocol (MCP).

## 1. Скиллы для получения достоверной информации (Reliability & Truth)

Для борьбы с галлюцинациями и обеспечения 100% достоверности стоит внедрить следующие паттерны:

### а) Deep Research & Fact-Checking
*   **Skill: `automated-fact-checking`**
    *   **Суть**: Интеграция с *Google FactCheck API* или создание специализированных агентов-фактчекеров (по паттерну AI-агента *Truth Sleuth*, который использует RAG по проверенным базам). Важно понимать, что Truth Sleuth — это скорее архитектурный паттерн специализированного ChatGPT-агента, а не готовый MCP-сервер, поэтому для него потребуется написать свой кастомный MCP или Skill.
    *   **Паттерн**: "Claim Extraction → Multi-source Verification → Veracity Reporting".
*   **Skill: `semantic-web-discovery` (Exa MCP)**
    *   **Суть**: Переход от ключевых слов к семантическому поиску. Exa позволяет искать "по смыслу", что критично для нахождения актуальной документации, которая еще не проиндексирована стандартными SEO-движками.

### б) Agentic RAG с "Self-Reflection"
*   **Паттерн**: Агент не просто читает файл, а оценивает его релевантность перед использованием.
*   **Рекомендация**: Создать `templates/verified-knowledge-skill`, где каждый шаг чтения сопровождается `<thought>` блоком с оценкой доверия к источнику.

## 2. Скиллы для развития проекта (Development & Architecture)

### а) Глубокая интеграция с Codebase (GitHub MCP)
*   **Skill: `repo-sentinel`**
    *   **Суть**: Автоматический мониторинг Issue и PR. Использование GitHub MCP для анализа изменений не только в коде, но и в архитектурных документах.
    *   **Цель**: Гарантировать, что новые скиллы не нарушают принципы `guides/`.

### б) Скиллы "Vibe Coding" и Автоматизации
*   **Skill: `environment-orchestrator` (Docker MCP)**
    *   **Суть**: В 2026 году Docker Desktop внедрил *MCP Toolkit* и *MCP Catalog*. Агент сам может поднимать изолированные контейнеры для тестирования `scripts/` внутри скилла, управляя инфраструктурой как кодом (Docker Compose для AI-стеков). Это полностью решает проблему безопасности и зависимости локальных окружений агентов.
*   **Skill: `automated-mcp-generator`**
    *   **Суть**: Использование паттернов, аналогичных подходу платформы Mintlify, которая научилась автоматически генерировать MCP-серверы из существующей документации. Скилл может анализировать API любого сервиса и автоматически генерировать для него OpenAPI-спецификацию в виде MCP-сервера.

## 5. Стратегия реализации Phase 3: Automated Quality & LLM-as-judge (На основе SOTA 2025-2026)

Исследование актуальных практик (`microsoft/llm-as-judge`, `langchain-ai/openevals`, кейсы production-уровня от ZenML и Slack) показало, что лучшие команды ушли от простых LLM-скриптов в сторону многоагентных систем с мощной фильтрацией и песочницами.

### а) Обновление системы эвалюации (`templates/skill-evals/`)
Переход от жестко заданных скриптов (типа старого `eval_code_grader.py`) к интеллектуальным метрикам:
1.  **Agent Trajectory Evaluation**: Оценка не только финального ответа, но и "траектории" агента (какие MCP-инструменты он вызывал и в каком порядке). Паттерны оценки: `strict` (строгий порядок), `unordered` (все нужные инструменты вызваны).
2.  **Песочницы для кода (Sandboxed Execution)**: Выполнение сгенерированного скриптами/скиллами кода в изолированных средах (например, E2B или Docker контейнеры). Это позволяет проверять Type Checking (Pyright/TypeScript) и работоспособность кода, а не просто парсить строки.
3.  **Специфичные LLM-метрики**: Использование `LLM-as-judge` для оценки конкретных параметров (Helpfulness, Groundedness, Toxicity, Hallucination) с обязательным требованием от LLM выдавать скоринг в строгом формате (JSON/Pydantic).

### б) LLM-as-judge для ревью PR в Skill-Factory
Автоматизированное код-ревью внедряется через GitHub Actions, но использует асинхронную многоагентную архитектуру:
1.  **Декомпозиция на микро-агентов (Parallel Generators)**: Вместо одного большого бота-ревьюера используются специализированные субагенты, работающие параллельно. Один ищет уязвимости безопасности, второй проверяет логику скилла, третий оценивает качество промптов.
2.  **Многоступенчатая фильтрация (Filtering Pipeline)**: Главная проблема AI-ревью — высокий уровень "шума" (ложных срабатываний). Все сгенерированные комментарии проходят через финального агента-судью, который отсеивает дубликаты, проверяет галлюцинации по "доказательствам" из логов и отклоняет неуверенные комментарии (Confidence thresholding).
3.  **Code Search Subagent (AST RAG)**: Чтобы не забивать контекст всем репозиторием, используется умный поиск, который по AST-дереву (Abstract Syntax Tree) находит только зависимые от изменённого PR файлы и передает их агенту-ревьюеру как контекст.
4.  **Модель смешивания (Model Mixing)**: Использование сильных сторон разных моделей одновременно (например, Claude 3.5 Sonnet для проверки кода и написания скриптов, а тяжелые модели семейства o1/o3 — для аудита сбоев в траекториях агентов).
