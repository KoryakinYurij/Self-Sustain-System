# Исследование Oh My Opencode: Уроки для создания скиллов

В ходе анализа репозитория `oh-my-opencode` были выявлены ключевые архитектурные решения и паттерны промптинга, которые делают этот плагин мощным инструментом оркестрации. Эти принципы можно напрямую применять при создании собственных скиллов и агентов.

## 1. Архитектура Агентов и Делегирование

Oh My Opencode использует строгую специализацию (11 специализированных агентов), где каждый агент имеет свой `AgentConfig` и четкую роль.
- **Sisyphus (Оркестратор):** Главный агент. Его главная мантра — **"Никогда не работай один, если доступны специалисты"**. Он классифицирует намерения, исследует кодовую базу и делегирует задачи через `task()` tool.
- **Prometheus (Планировщик):** Работает в "интервью-режиме", задавая уточняющие вопросы до написания кода, создавая строгий план.
- **Oracle (Консультант):** Read-only агент с высоким IQ (обычно GPT-5.2 или Claude 4.5 с thinking). Привлекается для ревью архитектуры или когда Sisyphus не может решить баг за 3 попытки.
- **Explore / Librarian:** Легковесные агенты-ищейки. Sisyphus запускает их **параллельно и в фоне** (`run_in_background=true`) для быстрого поиска по файлам и документации.

## 2. Анатомия Промптов (Опыт Sisyphus)

Промпты написаны с инженерной точностью и используют XML-тегирования для структурирования:

### а) Intent Gate (Врата намерений)
Агент обязан "вербализовать" (проговорить вслух) тип задачи перед тем, как что-либо делать.
> "I detect [research / implementation / fix] intent — [reason]. My approach: [explore → answer / plan → delegate]."
Это предотвращает галлюцинации и лишнюю работу.

### б) Строгие правила делегирования
Если агент делегирует задачу, он обязан предоставить исчерпывающий контекст из 6 пунктов:
1. **TASK:** Атомарная цель.
2. **EXPECTED OUTCOME:** Желаемый результат.
3. **REQUIRED TOOLS:** Белый список инструментов (ограничивает "разрастание" полномочий).
4. **MUST DO:** Обязательные требования.
5. **MUST NOT DO:** Что строго запрещено.
6. **CONTEXT:** Файлы и паттерны.

### в) Session Continuity (Непрерывность сессий)
Когда Sisyphus вызывает под-агента, он сохраняет `session_id`. Пи использовании под-агента повторно, он передает `session_id`, а не описывает задачу заново. Это сохраняет 70% токенов и полный контекст разговора.

### г) Failure Recovery (Восстановление после ошибок)
Встроен алгоритм "Что делать, если сломалось":
- После **3 неудачных попыток** фикса → СТОП. Откатить изменения. Спросить Oracle. Не помогло → Спросить пользователя.

### д) Оптимизация под конкретные LLM
Промпты динамически меняются в зависимости от модели:
- Для **Claude / Gemini / GPT** используются разные структуры. Например, для Gemini в конец промпта еще раз добавляются строгие правила верификации (чтобы бороться с "lost-in-the-middle" эффектом).

### е) Стратегия тестирования и QA (Agent-Executed QA)
В Oh My Opencode встроен принцип **обязательной** валидации. Даже если юнит-тесты не предусмотрены, агент-планировщик (Prometheus) требует включения *Agent-Executed QA Scenarios*.
Каждая задача в плане обязана иметь шаги, где агент сам запустит код и проверит его (например, через curl, playwright или запуском скрипта).

### ж) Управление длиной вывода (Incremental Write Protocol)
Чтобы избежать лимитов токенов при генерации длинных планов, применяется *Инкрементальный протокол записи* (`Incremental Write Protocol`):
1. Агент пишет структуру (skeleton) плана.
2. Итеративно добавляет блоки задач с помощью функции *Edit*, добавляя их до метки "Final Verification".
Это предотвращает зависания при генерации больших объемов текста.

### з) Ролевая спецификация ревьюеров (Anti-Nitpick)
Агент **Momus** (ревьюер планов) имеет строгий запрет на перфекционизм. Его задача — найти **только критические блокеры** (неверные пути к файлам, задачи без QA). Он имеет лимит в "Максимум 3 замечания на ревью", чтобы не создавать бесконечный цикл придирок к стилю или архитектуре. Главный принцип: *"Good enough is good enough. You are a blocker-finder, not a perfectionist."*

### и) Борьба с AI-Slop (Предотвращение over-engineering)
Агент **Metis** (пре-аналитик) анализирует намерения до создания плана, чтобы отсечь типичные галлюцинации LLM (AI-slop): добавление ненужных абстракций, избыточная генерация комментариев, раздувание скоупа за пределы запроса. Он заставляет явно прописывать секцию "MUST NOT DO" для исполнителя. 

### к) Date Awareness & Doc Discovery (Для агентов-ищеек)
Агент **Librarian**, специализирующийся на поиске во внешней среде, имеет механизм *Date Awareness*: он алгоритмически заставляет себя всегда указывать текущий год при web-поиске библиотек, чтобы отсекать устаревшие паттерны. Также, вместо слепого поиска, он сначала применяет *Doc Discovery* (парсинг `sitemap.xml` официальной документации), чтобы точно знать навигационную структуру до чтения страниц. Любые утверждения он обязан подтверждать GitHub Permalink'ами с указанием точных строк.

## 3. Выводы для создания скиллов

1. **Модульность важнее монолитности:** Лучше написать 3 узкоспециализированных скилла, которые можно вызывать параллельно, чем 1 огромный промпт, который делает все.
2. **Вербализация планов:** Заставляй агента в скилле проговаривать план до использования инструментов.
3. **Структура XML:** Используй `<Behavior_Instructions>`, `<Constraints>`, `<intent_verbalization>` — это сильно повышает послушность современных моделей.
4. **Управление инструментами:** Не давай под-агентам лишние инструменты. Ограничивай их возможности ради фокуса.
5. **Параллелизм:** Если скилл ищет информацию, настраивай его на асинхронный поиск.
