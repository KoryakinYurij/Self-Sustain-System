# Идеи из Codex Skill Creator для V3

Анализ промпта `skill-creator` от Codex Open AI выявил несколько выдающихся концепций инженерии контекста, которые стоит перенести в наш фреймворк Skill Factory V3.

## 1. Фреймворк «Степеней свободы» (Degrees of Freedom)
Вместо жестких правил делегирования для всех задач, вводится гибкая система в зависимости от хрупкости задачи:
*   **Высокая свобода (High freedom):** Открытое поле. Используются текстовые инструкции и эвристика (правила большого пальца). Подходит, когда решений много и подход сильно зависит от контекста.
*   **Средняя свобода (Medium freedom):** Шаблоны и псевдокод, скрипты с параметрами. Подходит, когда есть предпочтительный паттерн, но допустимы вариации.
*   **Низкая свобода (Low freedom):** Узкий мост над пропастью. Хрупкие, ломающиеся операции. Никакой свободы — только жесткие Bash/Python скрипты без вариаций, чтобы гарантировать консистентность.
*   **Внедрение:** При создании скилла разработчик сразу определяет уровень свободы задачи и выбирает инструмент (промпт в SKILL.md vs вызов готового скрипта).

## 2. Прогрессивное раскрытие через файловый роутинг (Progressive Disclosure Patterns)
Отказ от мега-файлов `SKILL.md`. Вместо этого основной файл скилла становится **маршрутизатором**.
*   Агент не подгружает инструкции для всех возможных вариантов рабочей среды (например, AWS, GCP, Azure одновременно). Он читает `SKILL.md`, понимает намерение пользователя и подгружает только нужный детализированный сценарий (например, `references/aws.md`).
*   **Внедрение:** Создать паттерн, где скилл работает как "оглавление" с гиперссылками, заставляя агента подтягивать дополнительные `.md` файлы только по мере углубления в узкую подзадачу, не перегружая основное контекстное окно.

## 3. Разделение References (справочники) и Assets (активы)
Жесткое разделение вспомогательных файлов на две категории для экономии токенов:
*   **`references/`**: Документация, которую агент **должен прочитать** и держать в уме (например, спецификации API, `schema.md`, шаблоны NDA).
*   **`assets/`**: Файлы, которые агент **должен использовать в аутпуте, но не обязан читать или парсить** (картинки, шаблоны `react-boilerplate`, шрифты, презентации).
*   **Внедрение:** Инструкция агентам не делать парсинг/чтение для файлов из папки `assets/`. Агент должен оперировать ими через скрипты или средства ОС (скопировать, переименовать, передать в компилятор) как с "черными ящиками".

## 4. Запрет на «Человеческую» документацию (Anti-Human Clutter)
Скилл создается строго для ИИ, а не для человека, листающего репозиторий на GitHub. ИИ не должен тратить вычислительные ресурсы на понимание истории версий или процесса установки.
*   Строгий запрет на наличие в папке скилла "человеческих" файлов: `README.md`, `CHANGELOG.md` или `INSTALLATION_GUIDE.md`.
*   **Внедрение:** Жесткое правило (или линтер) для скиллов, вычищающее любой контент и информацию о том, "как создавался этот скилл" — оставляя только чистую процессуальную информацию для выполнения работы.

## 5. Исполнение без предварительной загрузки в контекст (Execution without context loading)
*   Готовые скрипты в папке `scripts/` (например, `rotate_pdf.py`) предназначены для прямого вызова агентом, без необходимости предварительно читать их исходный код в свое контекстное окно (если не требуется отладка).
*   Агент воспринимает их как надежный инструмент с известными аргументами.
*   **Внедрение:** Упаковка детерминированной, повторяющейся логики в изолированные скрипты. Это отлично ложится на идею использования Headless CLI (Gemini CLI / Codex CLI) без замусоривания контекста исходниками этих скриптов.
