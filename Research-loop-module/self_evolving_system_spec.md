## 1) Что это за проект

**Self-Evolving System** — это модульная система, которая со временем улучшает собственную способность:

- писать и улучшать агентов;
- писать и улучшать скиллы;
- накапливать проверенные паттерны;
- опираться на качественные исследования и надёжные источники;
- превращать исследования в конкретные изменения и проверять эффект от внедрения.

На текущем этапе система не пытается быть полностью автономной. Сначала она строится как **управляемая модульная система** с первым рабочим контуром самоулучшения: `research-loop`.

## 3) Главная идея архитектуры

Система делится на несколько слоёв:

1. **Core**

   - описывает, что это за система;
   - фиксирует архитектурные решения;
   - задаёт правила взаимодействия модулей.

2. **Research Loop**

   - ищет качественные источники;
   - фильтрует и проверяет их;
   - извлекает полезные паттерны;
   - формулирует предложения по улучшению;
   - проверяет эффект после внедрения.

3. **Agents Lab**

   - пространство разработки агентов;
   - сюда приходят улучшения, связанные с архитектурой агентов, инструкциями, ролями, orchestration-паттернами и tool-routing.

4. **Skills Lab**

   - пространство разработки скиллов;
   - сюда приходят улучшения, связанные со структурой скиллов, шаблонами, критериями качества, анализом чужих удачных реализаций.

5. **Memory**

   - место, где сохраняются принятые паттерны;
   - сюда попадают не все идеи, а только те, которые прошли проверку и дали полезный эффект.

6. **Evaluations**

   - критерии оценки;
   - чеклисты;
   - способ понять, стало лучше или нет.

7. **Artifacts**

   - generated outputs;
   - собранные отчёты;
   - временные артефакты;
   - экспортированные материалы.

---

## 4) Неполезные крайности, которых надо избегать

Не надо:

- пытаться сразу строить полностью автономный self-improving AGI;
- создавать слишком много модулей раньше времени;
- складывать в память всё подряд без фильтра;
- копировать паттерны из источников без проверки применимости;
- смешивать research, evaluation и accepted memory в одну кучу;
- делать красивую документацию без рабочего цикла внедрения.

---

## 5) Первый рабочий MVP

Первый MVP = **один ручной завершённый цикл **``.

Цель цикла:

1. выбрать одну тему исследования;
2. собрать 3–5 качественных источников;
3. сделать короткий review по каждому;
4. сформулировать 1–3 proposal;
5. проверить хотя бы одно изменение;
6. зафиксировать эффект;
7. если сработало — перенести паттерн в `memory`.

### Первая рекомендуемая тема

`agent-instruction-patterns`

То есть:

- как сильные системы описывают инструкции агентам;
- какие паттерны повторяются;
- что из этого можно аккуратно перенести в `agents-lab`.

# System Blueprint

## Назначение

Self-Evolving System — это проект, который должен постепенно повышать качество собственных результатов через модульное улучшение.

Система улучшает не всё сразу, а отдельные блоки:
- подходы к разработке агентов;
- подходы к разработке скиллов;
- внутренние правила и шаблоны;
- накопление проверенных паттернов;
- исследовательские процессы.

## Архитектурный принцип

Каждый модуль должен быть улучшабелен отдельно.

Это означает:
- у каждого модуля должна быть понятная роль;
- улучшения должны приходить в модуль адресно;
- эффект изменений должен быть наблюдаем;
- удачные паттерны должны накапливаться;
- неудачные попытки тоже должны фиксироваться, чтобы не повторять их бесконечно.

## Базовые модули

### 1. Core
Содержит описание системы, правила и архитектурные решения.

### 2. Research Loop
Собирает, проверяет и превращает знания во внедряемые предложения.

### 3. Agents Lab
Место, где развиваются подходы к созданию агентов.

### 4. Skills Lab
Место, где развиваются подходы к созданию скиллов.

### 5. Memory
Хранилище принятых и проверенных паттернов.

### 6. Evaluations
Критерии и способы проверки, стало ли лучше.

### 7. Artifacts
Производные артефакты: отчёты, сборки, временные файлы.

## Первый приоритет

Первый приоритет — не расширение количества модулей, а запуск простого воспроизводимого `research-loop`, который:
1. берёт тему;
2. собирает качественные источники;
3. фильтрует их;
4. формулирует предложения;
5. проверяет их;
6. записывает результат.

## Ограничение качества

Желательно опираться в первую очередь на проверенные источники:
- OpenAI
- Anthropic
- Google
- хорошие технические репозитории и качественный проверенный код

Если источник слабый, это должно быть явно отмечено.

# ADR-0001 — Modular foundation for Self-Evolving System

## Статус
Accepted

## Контекст

Проект формируется как система, которая со временем должна улучшать собственную способность:
- разрабатывать агентов;
- разрабатывать скиллы;
- накапливать качественные паттерны;
- улучшать отдельные части себя на основе исследований и проверки эффекта.

На раннем этапе возник риск смешать в одну папку:
- разработку скиллов;
- разработку агентов;
- исследования;
- правила системы;
- память;
- артефакты.

Это делало бы систему трудной для расширения и анализа.

## Решение

Принято строить систему модульно.

Базовые модули:
- `core`
- `modules/research-loop`
- `agents-lab`
- `skills-lab`
- `memory`
- `evaluations`
- `artifacts`

Отдельный `prompts-lab` пока не вводится, так как соответствующие задачи временно закрываются через `agents-lab` и `skills-lab`.

## Последствия

### Плюсы
- проще улучшать модули по отдельности;
- понятнее, где искать правила, где исследования, где результаты;
- легче запускать первый MVP;
- меньше смешения сущностей.

### Минусы
- придётся поддерживать дисциплину границ между модулями;
- часть модулей на старте будет минимальной;
- возможна последующая реорганизация, когда появятся новые реальные потребности.

## Следствие для первого этапа

Первым рабочим контуром становится `modules/research-loop`, который должен дать системе первый повторяемый цикл самоулучшения.

# Module Interface Contract

Этот документ фиксирует минимальные ожидания от любого модуля системы.

## Каждый модуль должен иметь

1. понятную цель;
2. границы ответственности;
3. входы;
4. выходы;
5. понятный способ улучшения;
6. связь хотя бы с одним соседним модулем.

## Правила интеграции с Memory

- При добавлении нового accepted pattern или anti-pattern в `memory`, нужно явно указать, какой базовый шаблон в `agents-lab` или `skills-lab` должен быть изменён или что нужно учесть, чтобы паттерн начал работать по умолчанию. Это гарантирует, что знания применяются ("активная память"), а не оседают мертвым грузом.

## Минимальный шаблон описания модуля

- Название модуля
- Цель
- Что принимает на вход
- Что производит на выход
- Какие решения может менять
- По каким критериям понимается улучшение
- Какие артефакты создаёт
- Куда отправляет подтверждённые результаты

## Пример

### Research Loop
- Цель: находить качественные знания и превращать их в предложения по улучшению
- Вход: тема исследования, набор источников
- Выход: review, proposal, experiment result
- Может менять: подходы в agents-lab и skills-lab
- Критерии улучшения: предложения становятся конкретнее, полезнее и проверяемее
- Артефакты: source entries, reviews, proposals, experiment logs
- Подтверждённые результаты отправляет в: memory

# Pattern Promotion Policy

Этот документ описывает процесс перехода паттерна от эксперимента к рабочему шаблону.

## Когда применяется

После того как experiment получил статус `accepted`.

## Шаги продвижения (promotion)

### 1. Записать в memory
Создать запись в `memory/accepted-patterns.md` по шаблону.
Обязательно заполнить:
- `valid_for` (модель/архитектура)
- `synchronization target` (какой шаблон обновлять)
- `confidence` (high / medium / low)

### 2. Обновить целевой шаблон
Перейти в указанный `synchronization target` (напр. `agents-lab/templates/...` или `skills-lab/templates/...`) и внести изменение, чтобы паттерн начал применяться по умолчанию.

### 3. Отметить синхронизацию
В записи `accepted-patterns.md` отметить:
- `Статус синхронизации: [x] обновлено`

### 4. Закрыть experiment
В experiment entry добавить пометку, что promotion выполнен.

## Шаги отклонения (rejection)

### 1. Записать в anti-patterns
Создать запись в `memory/anti-patterns.md` по шаблону.
Обязательно указать причину отказа и ссылку на experiment.

### 2. Откатить изменения (если были)
Если proposal уже был внедрён до получения финального статуса — откатить к baseline.

### 3. Закрыть experiment
В experiment entry зафиксировать `rejected` и причину.

## Важно

- Promotion без обновления шаблона = незавершённый promotion.
- Rejection без записи в anti-patterns = потерянный опыт.
- Оба варианта должны заканчиваться закрытием experiment entry.
```

# Research Loop

`research-loop` — первый MVP-модуль Self-Evolving System.

Его задача:
1. находить качественные источники;

На старте модуль работает вручную или полу-вручную.

# Research Loop Workflow

## Source Trust Tiers

При сборе источников каждый из них должен быть отмечен уровнем доверия:

| Tier | Описание | Примеры |
|------|----------|---------|
| **T1** | Official docs / primary sources | Документация OpenAI, Anthropic, Google; первоисточники авторов |
| **T2** | Сильные инженерные write-ups | Технические блоги от практиков, подробные post-mortems |
| **T3** | Хорошие репозитории / примеры | Проверенный open-source код, сильные community-решения |
| **T4** | Exploratory / слабые сигналы | Twitter-треды, неподтверждённые идеи, мнения без данных |

Источники T4 допускаются, но должны быть **явно помечены** и не могут быть единственным основанием для proposal.

---

## Два режима работы

### Full mode (по умолчанию)
Полный цикл: source-entry → review-entry → proposal-entry → experiment-entry.
Используется для серьёзных изменений, затрагивающих шаблоны, архитектуру или рабочие контракты.

### Lightweight mode
Один файл `research-digest.md` для маленьких исследований.
Используется, когда мысль небольшая и не требует разведения по 4–5 отдельным файлам.
Шаблон: `templates/research-digest.md`

Если lightweight-исследование перерастает в серьёзный proposal — переключиться на full mode.

---

## Базовый цикл (Full mode)

### Шаг 1. Выбрать тему
Тема должна быть узкой и понятной.

Примеры:
- agent-instruction-patterns
- tool-routing-patterns
- skill-template-quality-signals
- evaluation-checklist-design

### Шаг 2. Собрать источники
Найти 3–5 качественных источников по теме.
Каждому источнику присвоить Trust Tier (T1–T4).

### Шаг 3. Сделать review по каждому источнику
Ответить кратко:
- почему источник заслуживает внимания;
- что именно в нём полезно;
- что из него нельзя переносить бездумно;
- к какому модулю относится потенциальная польза.

### Шаг 4. Сформировать proposal
Proposal должен быть конкретным.
Не «улучшить агента», а, например:
- ввести новый шаблон agent instructions;
- добавить обязательный раздел tool selection;
- изменить порядок reasoning/acting;
- усилить критерии проверки skill quality.

### Шаг 5. Провести маленький experiment
Проверить proposal на одном узком кейсе.
Эксперимент должен быть маленьким и наблюдаемым.

**Обязательно:** зафиксировать Baseline (поведение до изменения) и проверить 2–3 инвариантных кейса (не должны ухудшиться).

### Шаг 6. Зафиксировать outcome
Варианты:
- accepted
- rejected
- inconclusive
- needs-more-testing

### Шаг 7. Promote или reject
- Если accepted → перенести паттерн в `memory/accepted-patterns.md` и обновить связанный шаблон (см. `pattern-promotion-policy.md`).
- Если rejected → перенести в `memory/anti-patterns.md` с причиной отказа.
- Если inconclusive/needs-more-testing → оставить в `experiments/` с пометкой для повторного прогона.

---

## Definition of Done (одного цикла research-loop)

Цикл считается **завершённым**, когда:

- [ ] есть минимум 3 источника с присвоенными Trust Tiers;
- [ ] есть минимум 1 конкретный proposal;
- [ ] есть 1 experiment с зафиксированным baseline;
- [ ] есть финальный статус: accepted / rejected / inconclusive;
- [ ] принято решение: **promote** (→ memory + template update) / **reject** (→ anti-patterns) / **defer** (→ оставить с пометкой).

Без этих пунктов цикл не должен считаться закрытым. Это защита от зависания в бесконечном research.

---

## Rollback Rule

Если внедрённый proposal сломал поведение или дал негативный эффект:

1. **Откатить** изменение к состоянию до эксперимента.
2. **Записать** причину отката в experiment entry (поле Notes).
3. **Перенести** паттерн в `memory/anti-patterns.md` с пометкой "rolled back".
4. **Не удалять** experiment entry — он остаётся как документация неудачной попытки.

Система должна уметь не только улучшать, но и безопасно отступать.
```
