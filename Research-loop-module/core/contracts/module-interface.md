# Module Interface Contract

Этот документ фиксирует минимальные ожидания от любого модуля системы.

## Каждый модуль должен иметь

1. понятную цель;
2. границы ответственности;
3. входы;
4. выходы;
5. понятный способ улучшения;
6. связь хотя бы с одним соседним модулем.

## Правила интеграции с Memory

- При добавлении нового accepted pattern или anti-pattern в `memory`, нужно явно указать, какой базовый шаблон в `agents-lab` или `skills-lab` должен быть изменён или что нужно учесть, чтобы паттерн начал работать по умолчанию. Это гарантирует, что знания применяются ("активная память"), а не оседают мертвым грузом.

## Минимальный шаблон описания модуля

- Название модуля
- Цель
- Что принимает на вход
- Что производит на выход
- Какие решения может менять
- По каким критериям понимается улучшение
- Какие артефакты создаёт
- Куда отправляет подтверждённые результаты

## Пример

### Research Loop
- Цель: находить качественные знания и превращать их в предложения по улучшению
- Вход: тема исследования, набор источников
- Выход: review, proposal, experiment result
- Может менять: подходы в agents-lab и skills-lab
- Критерии улучшения: предложения становятся конкретнее, полезнее и проверяемее
- Артефакты: source entries, reviews, proposals, experiment logs
- Подтверждённые результаты отправляет в: memory
