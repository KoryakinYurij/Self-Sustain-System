Как бы я разложил это дальше.

### 1. Сначала не multi-agent, а multi-backend
Первая версия должна уметь:
- взять одну задачу из backlog
- собрать контекст по repo
- вызвать **один** CLI backend
- получить ответ/patch
- прогнать тесты
- сохранить reasoning + verdict
- либо принять изменение, либо откатить

Не “много агентов”, а **один loop с единым контрактом runner-а**.

Минимальный контракт runner-а по смыслу такой:

- `prepare(context, task)`
- `run(prompt, limits)`
- `collect_artifacts()`
- `extract_patch()`
- `extract_summary()`
- `extract_followup_tasks()`

Тогда Codex CLI, Gemini CLI и любой другой backend становятся просто реализациями одного интерфейса.

### 2. Роли надо разделить логически уже сейчас
Даже если физически сначала всё делает один CLI-инструмент, логически лучше разделить роли:

- **Researcher** — ищет контекст и ограничения
- **Builder** — предлагает изменение
- **Critic** — ищет риски и слабые места
- **Verifier** — гоняет тесты и checks
- **Governor** — решает: merge / reject / retry

Иначе получится плохая петля: один и тот же агент сам придумал, сам сделал, сам одобрил.

### 3. Для старта я бы не делал Jules опорным узлом
Здесь важно не выдумывать: в этой сессии у меня нет web-доступа, поэтому я не могу проверить на сегодня официальный статус, CLI-поведение и ограничения Gemini CLI / Codex CLI / Jules по официальным источникам.

Но на уровне архитектуры рабочая гипотеза такая:
- **Codex CLI** разумнее сделать первым `Builder` backend-ом, если цель — правки по репо, патчи, ветки, тесты.
- **Gemini CLI** удобно рассматривать как `Researcher/Critic`, если нужен широкий поиск вариантов, альтернатив и разбор решений.
- **Jules** я бы пока не делал центральным исполнителем loop-а, пока не будет подтверждён его стабильный режим именно как headless CLI звена в твоём пайплайне.

### 4. Самое важное — guardrails до “саморазвития”
Без них loop быстро превратится в churn-машину.

Нужны жёсткие правила:
- никакой записи в `main` напрямую
- только branch/worktree на цикл
- лимит на число файлов, LOC и длительность цикла
- отдельно учитывать “изменения продукта” и “изменения самого loop-а”
- агент не может сам себе засчитывать improvement
- merge только если:
  - тесты зелёные
  - есть machine-readable summary
  - есть human-readable rationale
  - есть измеримый критерий улучшения

И ещё одно правило особенно важно:
**запрет на бесконечное переписывание собственных промптов, логов и описаний без внешнего эффекта**.  
Иначе система будет “развиваться” только текстом о развитии.

### 5. Что делать следующим шагом practically
Я бы выбрал такой порядок:

**Шаг A.** Выделить `RunnerAdapter` и `LoopState`  
**Шаг B.** Подключить только один backend, лучше тот, который даёт самый предсказуемый patch workflow  
**Шаг C.** Завести `CycleLog` с обязательными полями:
- task
- context sources
- command used
- patch summary
- test results
- decision
- next hypothesis

**Шаг D.** Разрешить loop-у делать только 3 типа изменений:
- тесты
- документация dev/process
- узкие рефакторинги под существующие тесты

Не feature development. Не architecture rewrite. Не self-modifying core.

### 6. Где тут реальный “self-sustain”
Настоящая самоподдержка начинается не там, где агент “сам себя улучшает”, а там, где система умеет:
- замечать деградации
- накапливать гипотезы
- проверять изменения
- откатывать плохие решения
- вести историю причин

То есть ядро — это не LLM.  
Ядро — это **governed loop + evidence trail**.

Мой текущий рабочий тезис:
**тебе сейчас нужен не swarm, а disciplined single-loop with pluggable CLI brains.**

## Что даёт вертикальный run

Вертикальная папка выигрывает там, где нужно выполнить **один законченный цикл**: тема, источники, review, proposal, experiment, decision. Для CLI-агента это удобно, потому что меньше прыжков по репо, легче собрать контекст для текущей задачи, проще архивировать, откатывать и смотреть историю конкретного случая. Это очень хорошо бьётся и с вашими логами, где первый MVP определён как один завершённый `research-loop`, и с предложением из `structure_optimization_proposals.md`, где прямо указано, что горизонтальная раскладка размазывает одно исследование по нескольким каталогам.

Но у чисто вертикальной схемы есть слабое место: она плохо подходит как **долговременная база знаний**. Если accepted patterns, anti-patterns, шаблоны и rubrics будут жить только внутри run-папок, агенту потом придётся искать по множеству эпизодов одно и то же правило. Это хуже для повторного использования и хуже для RAG-подобной выборки релевантных кусочков. И ваши собственные документы уже это понимают: у вас есть отдельные `memory`, `evaluations`, contracts и promotion policy, то есть канонический слой у вас по смыслу уже горизонтальный.

## Что даёт горизонтальная структура + сильный skill

Горизонтальная схема хороша как **постоянный слой правил**: шаблоны, policy, memory, rubrics, чеклисты, module contracts. Тут как раз очень полезны хорошие project instructions и project-level agents. Anthropic официально рекомендует проектную память в `./CLAUDE.md`, возможность импортировать дополнительные файлы, а также project subagents как Markdown-файлы с YAML frontmatter. Это отличный аргумент в пользу того, что “сильная инструкция по проекту” реально нужна и реально работает.

Но здесь ломается другой кусок: если **весь операционный цикл** тоже оставить горизонтальным и надеяться, что skill всё свяжет, агенту придётся постоянно делать лишнюю навигацию: открыть тему, потом идти в `sources/`, потом в `reviews/`, потом в `proposals/`, потом в `experiments/`, а потом ещё помнить, что именно относится к текущему run. Это не невозможно, но для headless CLI-режимов это хуже по надёжности, особенно когда у агента есть доступ к shell, file read/write, web fetch и MCP, как у Gemini CLI. Google прямо делает акцент на инструментальности и предупреждает про осторожность с auto-approve / yolo и про необходимость ограничивать инструменты. Это значит, что тебе нужно уменьшать не только риск “плохих действий”, но и риск “потери нити”.

## Мой итоговый вердикт

Переезжать в полностью вертикальную структуру — тоже не советую.

Практически это значит:

Горизонтально оставить:
- `memory/`
- `evaluations/`
- `agents-lab/`
- `skills-lab/`
- общие policies / contracts / runbook

Вертикально добавить:
- `modules/research-loop/runs/` или `campaigns/`

  - useful → в `memory/accepted-patterns`

Это очень хорошо согласуется с вашей `pattern-promotion-policy`: результат эксперимента не должен оставаться только в trace, он должен либо перейти в memory и обновить шаблон, либо стать anti-pattern.
