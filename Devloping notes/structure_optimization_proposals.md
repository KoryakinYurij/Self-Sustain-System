# Предложения по SOTA-оптимизации структуры Self-Evolving System

Здесь зафиксированы предложения по улучшению и оптимизации базовой структуры (от 2026-03-11), чтобы сделать систему надежнее, лучше читаемой для LLM-агентов и масштабируемой.

## 1. Вертикальная группировка (Campaigns / Runs) вместо горизонтальной категоризации
**Текущая картина:** Файлы одного исследования размазаны по папкам: `sources/`, `reviews/`, `proposals/`, `experiments/`. Если работать над темой *agent-instruction-patterns*, придется прыгать между 5 разными директориями.
**Решение:** Сгруппировать всё по "кампаниям" или "эпизодам" (Episodic Memory подход).  
```text
modules/research-loop/
└─ campaigns/ (или runs / episodes)
   └─ 2026-03-11-agent-instruction-patterns/
      ├─ plan.md (вместо inbox/...)
      ├─ sources.md
      ├─ reviews.md
      ├─ proposal.md
      └─ experiment.md
```
**Почему это лучше:**
Ограниченное "окно контекста" (Context Window). Когда агент или человек хочет проанализировать всё исследование или переиспользовать его (как трейс в MLflow/LangSmith), ему нужна одна директория. Архивировать или откатывать (rollback) исследование так тоже намного проще — удаляется/перемещается одна папка, а не выискиваются 5 файлов по всему модулю.

## 2. Атомарная память (Atomic Memory) вместо монолитных файлов
**Текущая картина:** Утвержденные паттерны хранятся в огромных файлах `memory/accepted-patterns.md` и `memory/anti-patterns.md`. Со временем эти файлы раздуются до десятков тысяч строк (Context Bloat).
**Решение:** Сделать память директорией с отдельными файлами.
```text
memory/
├─ patterns/
│  ├─ agent-routing-by-tool.md
│  └─ minimal-skill-template.md
└─ anti-patterns/
   └─ unlimited-loop-without-fallback.md
```
**Почему это лучше:**
Это фундаментальное требование для векторных баз данных (Vector DB / RAG). Когда агент ищет, как лучше поступить, намного эффективнее подтянуть в промпт 3 маленьких релевантных файла `*.md`, чем "скармливать" один гигантский монолит, из которого он может выцепить галлюцинации.

## 3. Машинно-читаемые метаданные (YAML Frontmatter)
**Текущая картина:** Метаданные вроде `Trust Tier: T1` или `Target module: agents-lab` пишутся просто как текст внутри Markdown-разметки (в виде списков или таблиц).
**Решение:** Внедрить YAML Frontmatter в начале всех шаблонов (`templates/`).
```markdown
---
id: agent-instruction-pattern-01
status: accepted
trust_tier: T1
target_module: agents-lab
date: 2026-03-11
---
# Описание паттерна...
```
**Почему это лучше:**
Когда система разрастется до сотен элементов, человеку или внутреннему агенту понадобятся скрипты (или утилиты вроде `fd`/`yq`/язык Python), чтобы быстро собирать статистику: *"Сделай выборку всех принятых паттернов с уровнем T1 для skills-lab"*. С простым текстом это делать долго и больно, с YAML-шапкой — моментально.

## 4. Локализация (Scoping) для Артефактов
**Текущая картина:** На корневом уровне есть общая папка `artifacts/`.
**Решение:** Убрать ее из корня или перенести ответственность за артефакты внутрь самих модулей (например, `modules/research-loop/artifacts/`). Альтернативно — складывать промежуточные "выхлопы" скриптов прямо в папку `campaigns/2026-03-11-xxx/tmp_artifacts/`.
**Почему это лучше:**
Корневые директории "Misc" или "Артефакты" всегда превращаются в глобальную мусорку (namespace pollution), где лежат файлы непонятно от какого процесса. Пусть каждый модуль "мусорит" у себя — так понятнее, кому принадлежит артефакт.
