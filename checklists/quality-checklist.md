# ✅ Чеклист качества Agent Skill

> Проверьте каждый пункт перед установкой / распространением скилла.
>
> **Источник:** [Skill Authoring Best Practices — Checklist](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)

---

## Метаданные (Frontmatter)

- [ ] Поле `name` — kebab-case, 1-64 символа, unicode строчные буквенно-цифровые + `-`
- [ ] Поле `name` совпадает с именем директории
- [ ] Поле `description` не пустое, 1-1024 символа
- [ ] `description` написан от **третьего лица** (не "I can help" и не "You can use")
- [ ] `description` объясняет **ЧТО** делает и **КОГДА** использовать
- [ ] `description` **НЕ содержит шагов** workflow (только триггеры, НЕ «как»)
- [ ] `description` содержит конкретные **ключевые слова** для обнаружения
- [ ] Имя скилла — **герундий/действие** (`processing-pdfs`, не `pdf-helper`)

## Качество контента (SKILL.md body)

- [ ] Тело `SKILL.md` — **до 500 строк**
- [ ] Тело `SKILL.md` — простые workflow **< 150 слов**, сложные **< 500 слов**
- [ ] Скилл решает **одну задачу** (1 скилл = 1 задача, micro-skills > монолиты)
- [ ] Нет ссылок `@skills/name` — только текстовые отсылки (например, `Use the deploying-apps skill`)
- [ ] Тяжёлый контент вынесен в отдельные файлы (`references/`, `assets/`)
- [ ] Ссылки на файлы — **максимум один уровень** глубины от SKILL.md
- [ ] Справочные файлы длиннее 100 строк имеют **оглавление** (Contents)
- [ ] Progressive disclosure используется уместно
- [ ] Нет время-зависимой информации (или в секции "Old patterns")
- [ ] **Консистентная терминология** — одно понятие = одно слово
- [ ] Примеры **конкретные**, не абстрактные
- [ ] Workflows имеют **чёткие шаги**

## Инструкции

- [ ] Не объясняется то, что агент **уже знает** (AI знает что такое PDF)
- [ ] Уровень свободы **соответствует задаче** (fragile → specific, flexible → general)
- [ ] Нет **слишком многих вариантов** — один дефолт + escape hatch
- [ ] Для сложных задач есть **feedback loops** (validate → fix → retry)

## Скрипты и код

- [ ] Скрипты **решают проблему**, а не перекладывают на агента
- [ ] Нет **интерактивных промптов** — все аргументы через CLI
- [ ] Обработка ошибок — **явная и полезная**
- [ ] Нет "magic constants" — все значения обоснованы
- [ ] Зависимости **указаны и проверены** как доступные
- [ ] Скрипты документированы через `--help`
- [ ] Нет **Windows-style путей** — все forward slashes (`/`)
- [ ] Есть шаги **валидации/верификации** для критических операций
- [ ] Скрипты **идемпотентны** где возможно
- [ ] Есть `--dry-run` для деструктивных операций
- [ ] Вывод — **структурированный** (JSON предпочтительнее таблиц)
- [ ] Размер вывода — **предсказуемый** (summary по умолчанию)

## Кросс-платформенность

- [ ] Файл называется **`SKILL.md`** (заглавными буквами)
- [ ] Пути внутри скилла используют **forward slashes** (`/`)
- [ ] `name` содержит только **строчные буквы, цифры и дефисы**
- [ ] Поле `compatibility` заполнено если есть системные требования
- [ ] Протестировано в целевых платформах (Gemini CLI / OpenCode AI)

## Тестирование

- [ ] Создано **минимум 3 тестовых сценария** использования
- [ ] Протестировано с **реальными задачами** пользователей
- [ ] Обнаружение (skill routing) работает корректно — скилл активируется по нужным запросам
- [ ] Фидбек от команды учтён (если применимо)

### Baseline Testing (TDD для скиллов)

Перед написанием скилла проведите baseline-тест:

1. **Без скилла:** попроси агента решить задачу *без* скилла → зафиксируй ошибки и рационализации
2. **С скиллом:** напиши скилл, закрывающий именно эти дыры → проверь снова
3. **Дельта:** сравни результаты — скилл должен устранять конкретные проблемы, а не добавлять шум

### 3 уровня тестирования (обязательны для production-ready)

| Уровень | Как | Когда |
|---------|-----|-------|
| **Manual** | Тест в Claude.ai / Gemini CLI с реальными запросами | При каждом изменении |
| **Scripted** | Серия промптов в Claude Code / терминале | После стабилизации |
| **Programmatic** | Evals через API (code graders + LLM-as-judge) | Перед деплоем |

- [ ] Baseline-тест проведён (без скилла vs с скиллом)
- [ ] Manual-тестирование пройдено
- [ ] Scripted-тестирование пройдено (для production-ready)

> 📎 **Anthropic Guide**: каждый уровень обязателен перед тем, как скилл считается production-ready.

## Метрики качества

Для каждого скилла отслеживайте:

| Метрика | Что измеряет | Как измерить |
|---------|-------------|-------------|
| **Trigger rate** | Как часто агент правильно активирует скилл | 10 релевантных запросов → сколько раз активировался? |
| **Tool calls count** | Эффективность (меньше = лучше) | Подсчитай вызовы инструментов при выполнении |
| **Token consumption** | Стоимость одного выполнения | API logs или расчёт по длине SKILL.md + references |
| **False trigger rate** | Активация на нерелевантных запросах | 5 нерелевантных запросов → сколько ложных срабатываний? |

- [ ] Trigger rate > 80% на релевантных запросах
- [ ] False trigger rate < 10%
- [ ] Token consumption в пределах ожиданий

> 📎 **Anthropic Guide**: без метрик невозможно отличить «работающий» скилл от «вроде работает».

## Быстрая валидация структуры

**Обязательный шаг перед деплоем** — запустите CLI-валидатор:

```bash
# Клонировать и установить skills-ref (Python)
git clone https://github.com/agentskills/agentskills.git
cd agentskills/skills-ref
pip install -e .

# Валидация
skills-ref validate path/to/my-skill
```

Валидатор проверяет:
- ✅ Корректный YAML frontmatter
- ✅ `name` совпадает с директорией
- ✅ `description` в пределах 1-1024 символов
- ✅ Структура директории соответствует спецификации

- [ ] `skills-ref validate` пройден без ошибок

> ⚠️ `skills-ref` — reference-библиотека (demonstration, не production).

> Источник: [agentskills.io/specification — Validation](https://agentskills.io/specification#validation)
