# План Обновления: V2 (Agentic Framework)

Этот документ собирает идеи и паттерны для грядущего масштабного обновления базы знаний. Мы переходим от формата "технической документации по написанию Markdown" к формату "психологического фреймворка для надежного программирования агентов".

## 🎯 Цель обновления
Сделать так, чтобы создаваемые по нашим гайдам скиллы учитывали особенности мышления LLM: их склонность к "срезанию углов" (рационализации), проблемы с удержанием контекста и оптимальные форматы восприятия логики.

## 📝 Бэклог задач на внедрение

### 1. Claude Search Optimization (CSO)
* **Где менять:** `guides/02-writing-effective-skills.md`, шаблоны.
* **Что сделать:** 
  * Ввести строгий запрет: `description` **никогда** не должен описывать workflow или шаги.
  * `description` должен содержать только *триггеры* (симптомы, условия, когда надо применять).
  * Объяснить, почему шорткаты в описании приводят к игнорированию тела `SKILL.md`.
  * 📎 **agentskills.io**: `name` обязан совпадать с именем директории; `description` 1-1024 символов, насыщен ключевыми словами для discovery.

### 2. Защита от рационализации (Anti-rationalization) и Red Flags
* **Где менять:** `guides/04-anti-patterns.md`, шаблоны.
* **Что сделать:**
  * Описать, как агенты под давлением лимитов или лени пытаются пропускать шаги (особенно ревью или тесты).
  * Во все шаблоны добавить обязательную секцию `## Red Flags`.
  * Инструктаж по явному закрытию лазеек: нужно не только говорить "делай", но и "ни в коем случае не делай иначе".

### 3. Оркестрация (Subagent-Driven Development)
* **Где менять:** Создать новый шаблон `templates/subagent-orchestration-skill/`, дополнить `guides/05-advanced-patterns.md`.
* **Что сделать:**
  * Описать паттерн "Исполнитель → Ревьюер ТЗ → Ревьюер Кода".
  * Объяснить важность порядка (сначала spec compliance, только потом code quality).
  * Предоставить базовые `prompt` шаблоны для запуска изолированных субагентов.

### 4. Графы вместо if/else (DOT-диаграммы)
* **Где менять:** `guides/02-writing-effective-skills.md`.
* **Что сделать:**
  * Рекомендовать использовать Graphviz DOT (` ```dot `) для деревьев принятия решений.
  * Объяснить, что агенты лучше понимают математически строгие графы, чем размытые текстовые `if/else`, особенно в сложных workflow.

### 5. Лимиты токенов и запрет @-ссылок
* **Где менять:** `guides/02-writing-effective-skills.md`, `checklists/quality-checklist.md`.
* **Что сделать:**
  * Ввести жесткие лимиты на длину `SKILL.md`: простые workflow < 150 слов, сложные < 500 слов. Всё остальное — в `references/`.
  * Ввести запрет на использование `@skills/name` внутри текстов скиллов, так как это заставляет агента выгружать чужой файл в контекст. Нужно использовать текстовые отсылки (например, `Use tool-name`).
  * 📎 **Anthropic Guide**: явное правило «1 скилл = 1 задача». Micro-skills, связанные цепочкой, надёжнее одного монолита.

### 6. Глагольное именование (Verb-first Naming)
* **Где менять:** `guides/01-skill-anatomy.md`.
* **Что сделать:**
  * Задать стандарт именования через активное действие/герундий (например, `creating-skills`, `systematic-debugging` вместо `skill-creator` или `debugging-guide`).

### 7. TDD для скиллов (Baseline Testing)
* **Где менять:** `checklists/quality-checklist.md`.
* **Что сделать:**
  * Добавить шаг тестирования "Baseline": попроси агента решить задачу *без* скилла → зафиксируй ошибки и рационализации → напиши скилл, закрывающий именно эти дыры → проверь снова.
  * 📎 **Anthropic Guide**: формализовать 3 уровня: manual (Claude.ai) → scripted (Claude Code) → programmatic (API). Каждый уровень обязателен перед тем, как скилл считается production-ready.

### 8. CLI-валидация SKILL.md (`agentskills.io`)
* **Где менять:** `checklists/quality-checklist.md`.
* **Что сделать:**
  * Добавить шаг `skills-ref validate` (CLI-валидатор из agentskills.io) в чеклист качества.
  * Автоматическая проверка формата перед деплоем: frontmatter, совпадение `name` с директорией, длина `description`.
  * Не заменяет ручной ревью, но отсекает тривиальные ошибки.

### 9. Executable Scripts (`scripts/`) — детерминированные операции
* **Где менять:** Новый шаблон `templates/executable-skill/`, дополнить `guides/02-writing-effective-skills.md`.
* **Что сделать:**
  * Создать гайд: **когда** оборачивать логику в `scripts/` (Python/Bash), а не полагаться на LLM.
  * Правило: если операция критическая (деплой, удаление, запись в БД, PII-редакция) — только скрипт. LLM вызывает скрипт, а не выполняет логику сам.
  * Шаблон: `scripts/run.py` + `scripts/validate.sh` + `requirements.txt` с изолированными зависимостями.
  * 📎 **Anthropic Guide**: «Scripts are deterministic, fast, and testable — use them for critical operations instead of hoping the LLM gets it right.»

### 10. Обязательный Hand-edit сгенерированных скиллов
* **Где менять:** `guides/02-writing-effective-skills.md`.
* **Что сделать:**
  * Добавить секцию «Post-Generation Review» — если агент сгенерировал скилл, **обязательно** отредактируй руками.
  * Типовые проблемы LLM-скиллов: многословность body, нечёткие trigger-ы в `description`, избыточные шаги, отсутствие edge cases.
  * Чеклист: description ≠ workflow → только trigger-ы? body < лимита? скрипты изолированы?

### 11. Метрики качества скилла
* **Где менять:** `checklists/quality-checklist.md`.
* **Что сделать:**
  * Ввести конкретные метрики для оценки скилла:
    * **Trigger rate** — как часто агент правильно активирует скилл при релевантном запросе.
    * **Tool calls count** — количество вызовов инструментов (меньше = эффективнее).
    * **Token consumption** — потребление токенов на одно выполнение скилла.
  * Для каждой метрики — описать как измерить (manual prompting / API logs).
  * 📎 **Anthropic Guide**: без метрик невозможно отличить «работающий» скилл от «вроде работает».

### 12. Evals-as-Code (LLM-as-Judge + Code graders)
* **Где менять:** Новый шаблон `templates/skill-evals/`, дополнить `guides/05-advanced-patterns.md`.
* **Что сделать:**
  * Создать шаблон eval-скрипта для тестирования скилла:
    * **Code-based grader**: скрипт проверяет детерминированный выход (JSON Schema, exit code, наличие файлов).
    * **LLM-as-judge grader**: другая модель оценивает open-ended качество по рубрике.
  * Интеграция в CI/CD: evals запускаются автоматически при каждом изменении скилла.
  * Разница с п.7 (TDD): TDD — это процесс *создания* скилла. Evals — *непрерывный мониторинг* качества после деплоя.
  * 📎 **Anthropic Evals**: code graders + model graders + human graders, интегрированные в pipeline.
